# Data Lifecycle and Storage Architecture

## 1. Purpose

This document defines the v2 data lifecycle and storage architecture for the market scanner.

It exists to prevent raw source data, program-ready input data, generated analytical output, Decision Engine output, and reporting output from being mixed together.

The market scanner must preserve traceability from source data to normalized inputs, analytical classifications, Decision Engine records, and reporting communication.

This is a PM, analyst, and architecture document. It is not an implementation sprint and does not define code behavior by itself.

## 2. Core Principle

The v2 data lifecycle is a staged pipeline:

```text
external source
  -> raw source data
  -> normalized program-ready input
  -> analytical classification output
  -> Decision Engine output
  -> reporting output
```

Each stage has a different purpose and authority level.

The same CSV must not act as raw evidence, normalized input, generated analysis, and report output at the same time.

## 3. Mandatory Separation

The following separation is mandatory:

```text
raw source data != normalized input != generated output != report
```

Definitions:

- Raw source data is evidence captured from a source.
- Normalized input is program-ready data derived from raw source data and shaped to v2 contracts.
- Generated output is the result of scanner, analysis, classification, pipeline, or Decision Engine execution.
- Reporting output is communication derived from approved records and must not become input authority.

## 4. Data Lifecycle Stages

### 4.1 External Source

External source means an origin outside the v2 runtime.

Examples may include:

- SEC filings;
- broker exports;
- manually curated portfolio files;
- market data providers;
- company fundamentals providers;
- local user-maintained source files.

External source data is not trusted merely because it exists. It must pass through source-data governance before use.

### 4.2 Raw Source Data

Raw source data is the preserved capture of external data.

Purpose:

- preserve source evidence;
- support auditability;
- preserve provenance;
- allow reprocessing later;
- prevent silent transformation from hiding source problems.

Rules:

- keep values as close as practical to the original source;
- store source identifier, retrieval timestamp, source period, ticker or entity identifier, provider name, and provenance where available;
- do not silently convert missing numeric values to zero;
- do not silently interpolate periods;
- do not use raw source files directly as program-ready inputs unless an approved contract explicitly allows it.

Raw data may be local-only, private, or ignored by Git depending on the source and sensitivity.

### 4.3 Normalized Program-Ready Input

Normalized input is derived from raw source data and shaped to v2 data contracts.

Purpose:

- provide stable, predictable input for v2 code;
- standardize column names, data types, source metadata, missing-value policy, and row identity;
- decouple provider-specific formats from the internal program model.

Rules:

- every normalized dataset must have an approved contract;
- every normalized row must preserve source traceability;
- missing values must remain explicit;
- normalized input must not contain final actions;
- normalized input must not contain portfolio allocation instructions;
- normalized input must not be overwritten by reporting or generated outputs.

Normalized input is the main data shape the v2 application should eventually consume.

### 4.4 Analytical Classification Output

Analytical classification output is generated by upstream analytical layers.

Examples:

- source-data readiness records;
- financial health classification;
- business quality classification;
- growth profile classification;
- profitability classification;
- valuation context classification;
- timing context classification;
- portfolio exposure classification.

Rules:

- classification is upstream;
- classification is descriptive;
- classification must preserve uncertainty;
- classification must not create final actions;
- classification must not create allocation, tradeability, execution, conviction, urgency, or hidden filtering authority.

### 4.5 Decision Engine Output

Decision Engine output is the only place where final decision authority may eventually exist.

Rules:

- Decision Engine is the only allocation, execution, arbitration, and final-action authority;
- upstream data and classification layers must not bypass the Decision Engine;
- Decision Engine output must remain traceable to upstream records;
- Decision Engine output must not be rewritten by reporting.

Current v2 status: the Decision Engine scaffold emits deterministic review-only records.

### 4.6 Reporting Output

Reporting output communicates records.

Rules:

- reporting communicates only;
- reporting must not create, alter, suppress, prioritize, filter, reinterpret, or override decisions;
- reporting output must not become source-of-truth input;
- reporting output must not be used as a raw or normalized data source.

## 5. Recommended Directory Model

The preferred v2 directory model is:

```text
data/
  raw/
  normalized/
  fixtures/
    v2/
  generated/
  local/
reports/
```

The exact directory skeleton must be implemented in a later Codex/local step after approval.

### 5.1 `data/raw/`

Purpose:

- original or near-original source captures;
- provider-specific or source-specific evidence;
- audit and reprocessing base.

Expected content examples:

- raw SEC captures;
- raw broker exports;
- raw provider responses;
- raw manually supplied source files.

Governance:

- not automatically source-of-truth for v2 code;
- may be private or ignored depending on source;
- must preserve provenance;
- must not be silently edited into program-ready shape.

### 5.2 `data/normalized/`

Purpose:

- program-ready v2 input datasets;
- contract-compliant CSV files;
- stable schemas consumed by v2 code.

Expected content examples:

- normalized fundamentals input;
- normalized portfolio transaction input;
- normalized source-data readiness input;
- normalized market context input if later approved.

Governance:

- must be derived from approved raw/source data or approved manual input;
- must conform to `docs/active/data_contracts.md` or a later approved contract;
- must preserve row identity and source traceability;
- must not contain final decisions or reporting text.

### 5.3 `data/fixtures/v2/`

Purpose:

- small synthetic deterministic test fixtures;
- contract-focused test inputs;
- no real portfolio or provider source-of-truth.

Governance:

- fixtures are allowed to be tracked;
- fixtures must remain synthetic unless explicitly approved otherwise;
- fixtures must not be confused with production source data.

### 5.4 `data/generated/`

Purpose:

- generated pipeline outputs;
- analytical classification outputs;
- Decision Engine output records;
- logs or review outputs if approved.

Governance:

- generated output is not source-of-truth;
- generated output must not be used as raw input;
- generated output must not be committed automatically to `main` unless explicitly approved;
- generated output should generally be reproducible from raw plus normalized input and code.

### 5.5 `data/local/`

Purpose:

- local-only private data;
- credentials-free user-specific working data;
- data that must not be committed.

Governance:

- should generally be ignored by Git;
- must not become canonical input without explicit approval;
- may be used for local experiments only when clearly separated.

### 5.6 `reports/`

Purpose:

- communication outputs;
- human-readable summaries;
- delivery payloads if later approved.

Governance:

- reports are not source data;
- reports are not normalized input;
- reports must not be read back into the pipeline as authority;
- reports must not create decisions.

## 6. Existing Legacy Data Paths

The repository currently contains historical data paths that predate the v2 lifecycle.

Examples include:

```text
data/processed/
data/portfolio/
data/logs/
data/watchlist/
reports/daily/
```

These paths must be treated as legacy or reapproval-required until explicitly classified by a future data cleanup and migration step.

No existing generated legacy CSV may become v2 source-of-truth merely because it exists in the repository.

## 7. Manual Portfolio Data

Portfolio data needs special treatment.

A manually maintained portfolio transaction source may eventually become approved input if it is:

- explicitly identified as manual source-of-truth;
- contract-defined;
- separated from generated portfolio review outputs;
- traceable;
- protected from automatic overwrite.

Generated portfolio files such as review outputs must not be confused with manual source records.

Expected future separation:

```text
manual portfolio source -> normalized portfolio input -> portfolio classification output -> Decision Engine
```

RESET-10C formalizes this boundary in `docs/active/portfolio_source_of_truth.md`.

Manual portfolio source transactions and manual portfolio source positions are the only portfolio source-of-truth roles approved by RESET-10C.

Generated portfolio review, generated portfolio intelligence, reporting display input, Telegram message text, and report output must not become portfolio source data.

Portfolio display fields needed by Telegram, including profit/loss, current price, target price, and action/status, must be supplied by approved upstream records. Reporting and Telegram may format those fields but must not calculate them or write them back as source truth.

## 8. Source-Data Readiness

Source-data readiness describes whether data is usable for the next step.

It may describe:

- available data;
- missing data;
- partial data;
- stale data;
- review-required data;
- source provenance;
- missing fields;
- period coverage;
- retrieval metadata.

It must not describe investment quality.

Source-data readiness is a data governance state, not a buy/sell/hold signal.

## 9. Missing Data Policy

Missing data must remain explicit across all lifecycle stages.

Mandatory rules:

- missing numeric values must not be converted to zero;
- missing text values must not be silently filled with plausible guesses;
- missing periods must not be silently interpolated;
- unavailable fields must be marked as missing, partial, stale, unavailable, or review-required;
- downstream layers must preserve uncertainty when relevant.

This protects the system from false precision and accidental recommendations.

## 10. Tracking and Git Policy

Future implementation must decide which data paths are tracked, ignored, or local-only.

Default policy:

- synthetic fixtures may be tracked;
- canonical documentation is tracked;
- raw private/provider data should usually not be tracked unless explicitly sanitized and approved;
- generated outputs should not be automatically committed to `main` unless explicitly approved;
- local user data should remain local and ignored;
- production-like generated reports should not become source-of-truth.

A later implementation step must update `.gitignore` and directory placeholders if needed.

## 11. Provider and SEC Data

SEC or provider data must follow the same lifecycle:

```text
provider/SEC source -> raw capture -> normalized input -> classification -> Decision Engine -> reporting
```

Rules:

- no live source integration without approval;
- no provider cache becomes canonical input automatically;
- raw source captures must preserve provenance;
- normalized inputs must conform to contracts;
- stale, partial, or missing data must remain visible.

## 12. Runtime Read Rules

Future v2 code should eventually read from approved normalized inputs and fixtures.

It should not read directly from:

- reports;
- generated legacy outputs;
- legacy `data/processed/` files;
- local/private paths;
- raw provider captures without a normalization contract;
- historical archive documents.

Exceptions require explicit documentation and tests.

## 13. Runtime Write Rules

Future v2 code may eventually write generated outputs only to approved generated-output locations.

Rules:

- writes must not overwrite raw source evidence;
- writes must not overwrite normalized input without an explicit transformation step;
- writes must not push directly to `main` through automation unless explicitly approved;
- writes must preserve deterministic and auditable behavior where possible;
- reports must remain communication outputs.

## 14. Data Contract Requirements

Every normalized input dataset should eventually define:

- dataset name;
- lifecycle stage;
- owner;
- source path;
- allowed storage location;
- row identity;
- required fields;
- optional fields;
- missing-value policy;
- source provenance fields;
- freshness policy;
- validation rules;
- whether the dataset may be tracked in Git.

## 15. Fundamentals Raw-to-Normalized Boundary

RESET-10D formalizes the fundamentals raw-to-normalized boundary.

The approved boundary is:

```text
raw source capture
-> normalized fundamentals input
-> source-data readiness
-> downstream classification and communication inputs
```

Raw source capture is evidence. Normalized fundamentals input is program-ready input. Source-data readiness is a data governance state. Generated fundamentals outputs and reporting outputs are downstream artifacts and must not overwrite raw or normalized inputs.

The boundary does not authorize provider ingestion, SEC or EDGAR ingestion, file loading, file writing, runtime normalization, financial scoring, or Decision Engine behavior.
