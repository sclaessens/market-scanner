## Terminology Corrections

Corrected active operational documentation so that:

- scanner is described as discovery/classification, not allocation selection
- validation is described as structure classification, not tradeability gating
- context is described as leadership and relative-strength classification, not allocation eligibility
- watchlist is described as timing-state tracking, not execution instruction generation
- portfolio is described as exposure/risk-state modelling, not BUY/SELL authority
- Decision Engine is the only owner of final actions, tradeability, conviction, allocation priority, and capital allocation
- reporting is described as communication/presentation only

Legacy terms retained only when clearly contextualized as forbidden, historical, archival, research, or Decision Engine-owned:

- `tradeable_setup`
- `context_tradeable`
- `BUY NOW`
- `SET LIMIT BUY`
- `SET STOP BUY`
- `urgency`
- `actionable`
- `filtering-first`
- `execution gating`

## Schema Corrections

Active technical and functional docs were aligned to governance-clean runtime outputs.

Validation schema:

- primary contract is `structure_state`
- `structure_reason` is the governance-clean reason field
- `valid_setup` remains documented only as a deprecated compatibility alias for structure coherence
- validation documentation no longer describes tradeability or allocation eligibility

Context schema:

- context outputs are leadership and relative-strength classification fields
- `context_tradeable` and `context_tradeable_reason` are not active context outputs

Decision output schema:

- current runtime output is documented as `data/processed/final_decisions.csv`
- Decision Engine may emit `final_action`, `tradeability`, `conviction`, and `allocation_priority`
- those fields are not permitted as upstream layer outputs

Reporting schema:

- reporting may format Decision Engine output
- reporting may not create, reinterpret, hide, or prioritize allocation decisions independently
